## 2. 系统角色与详细用例

### 2.1 角色定义

```
普通用户 (USER)      → 提交工单、查看进度、接收通知
客服人员 (AGENT)     → 处理工单、查看AI建议、回复用户
管理员   (ADMIN)     → 分配工单、查看统计数据、管理用户
```

### 2.2 用例详细说明

#### 普通用户

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 注册/登录 | 邮箱+密码，JWT Token鉴权 | P0 |
| 提交工单 | 填写标题、描述、上传附件（可选）；提交后AI自动分类+优先级 | P0 |
| 查看工单列表 | 我的工单，按状态筛选 | P0 |
| 工单详情 | 查看处理进度、客服回复记录 | P0 |
| 实时通知 | WebSocket推送：工单状态变更、新回复 | P1 |
| 评价工单 | 工单关闭后可打分（1-5星）+ 文字评价 | P2 |

#### 客服人员

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 查看待处理工单 | 只看分配给自己的，按优先级排序 | P0 |
| 查看工单详情+AI建议 | 打开工单时自动触发：①AI生成回复草稿 ②展示相似历史工单 | P0 |
| 回复工单 | 采纳AI草稿或手动编写，发送后通知用户 | P0 |
| 更新工单状态 | PROCESSING → RESOLVED | P0 |
| 查看历史处理记录 | 我处理过的工单统计 | P2 |

#### 管理员

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 分配工单 | 将PENDING工单指派给指定客服 | P0 |
| 数据大盘 | 今日工单量、平均响应时长、各类型占比、AI采纳率 | P1 |
| 用户管理 | 查看用户列表、禁用账号 | P1 |
| 客服管理 | 查看客服工作量、绩效 | P2 |

## 3. 数据库设计

> 所有表统一存放在同一个 PostgreSQL 15 数据库中，结构化数据与向量数据共库，无需维护两个数据源。

### 3.1 初始化

```sql
-- 启用 pgvector 扩展（仅需执行一次）
CREATE EXTENSION IF NOT EXISTS vector;
```

### 3.2 表结构

#### `users` 用户表

```sql
CREATE TABLE users (
    id          BIGSERIAL PRIMARY KEY,
    username    VARCHAR(50)  NOT NULL UNIQUE,
    email       VARCHAR(100) NOT NULL UNIQUE,
    password    VARCHAR(255) NOT NULL,   -- BCrypt加密
    role        VARCHAR(10)  NOT NULL DEFAULT 'USER'
                    CHECK (role IN ('USER','AGENT','ADMIN')),
    status      VARCHAR(10)  NOT NULL DEFAULT 'ACTIVE'
                    CHECK (status IN ('ACTIVE','DISABLED')),
    avatar_url  VARCHAR(255),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `ticket` 工单表

```sql
CREATE TABLE ticket (
    id              BIGSERIAL PRIMARY KEY,
    title           VARCHAR(200) NOT NULL,
    description     TEXT NOT NULL,
    category        VARCHAR(20) CHECK (category IN ('TECHNICAL','BILLING','COMPLAINT','OTHER')),  -- AI填充
    priority        VARCHAR(10) CHECK (priority IN ('LOW','MEDIUM','HIGH','URGENT')),             -- AI填充
    status          VARCHAR(20) NOT NULL DEFAULT 'PENDING'
                        CHECK (status IN ('PENDING','ASSIGNED','PROCESSING','RESOLVED','CLOSED')),
    submitter_id    BIGINT NOT NULL REFERENCES users(id),
    assignee_id     BIGINT REFERENCES users(id),           -- 客服
    ai_classified   BOOLEAN DEFAULT FALSE,                  -- 是否已AI分类
    rating          SMALLINT CHECK (rating BETWEEN 1 AND 5),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    resolved_at     TIMESTAMPTZ
);
```

#### `reply` 回复表

```sql
CREATE TABLE reply (
    id          BIGSERIAL PRIMARY KEY,
    ticket_id   BIGINT NOT NULL REFERENCES ticket(id),
    author_id   BIGINT NOT NULL REFERENCES users(id),
    content     TEXT NOT NULL,
    is_ai_draft BOOLEAN DEFAULT FALSE,   -- 是否来自AI草稿
    ai_adopted  BOOLEAN DEFAULT FALSE,   -- 客服是否直接采纳AI内容
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `notification` 通知表

```sql
CREATE TABLE notification (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT NOT NULL REFERENCES users(id),
    type        VARCHAR(20) NOT NULL
                    CHECK (type IN ('STATUS_CHANGE','NEW_REPLY','ASSIGNED')),
    ticket_id   BIGINT NOT NULL REFERENCES ticket(id),
    message     VARCHAR(500) NOT NULL,
    is_read     BOOLEAN DEFAULT FALSE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `ticket_embedding` 向量表

```sql
CREATE TABLE ticket_embedding (
    id              BIGSERIAL PRIMARY KEY,
    ticket_id       BIGINT NOT NULL UNIQUE REFERENCES ticket(id),
    content_text    TEXT NOT NULL,       -- 原始文本（title + description）
    embedding       vector(1536),        -- text-embedding-3-small 维度
    category        VARCHAR(20),
    solution_text   TEXT,               -- 最终解决方案（resolved后填入）
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- HNSW 向量索引（余弦相似度）
CREATE INDEX ON ticket_embedding USING hnsw (embedding vector_cosine_ops);
```

## 4. AI 功能详细设计

### 4.1 工单自动分类（提交时触发）

**触发时机**：用户提交工单后，异步执行（不阻塞响应）

**流程**：

```
用户提交工单
    │
    ▼
保存到 MySQL（status=PENDING, ai_classified=0）
    │
    ▼
异步发送给 DeepSeek（分类 Prompt）
    │
    ▼
解析返回的 JSON → 更新 ticket.category + ticket.priority
    │
    ▼
ai_classified = 1，同步写入 ticket_embedding（向量化存储）
```

**分类 Prompt 模板**：

```
你是一个客服工单分类助手。请根据以下工单内容，返回JSON格式的分类结果。

工单标题：{title}
工单描述：{description}

请严格按照以下JSON格式返回，不要输出任何其他内容：
{
  "category": "TECHNICAL|BILLING|COMPLAINT|OTHER",
  "priority": "LOW|MEDIUM|HIGH|URGENT",
  "reason": "一句话说明分类依据"
}

分类规则：
- TECHNICAL：涉及功能故障、无法登录、系统错误
- BILLING：涉及费用、退款、账单问题
- COMPLAINT：投诉服务态度或处理结果
- OTHER：不属于以上类别
优先级判断：URGENT=服务完全不可用；HIGH=主要功能受影响；MEDIUM=部分功能受影响；LOW=咨询类
```

---

### 4.2 AI 回复建议（客服打开工单时触发）

**触发时机**：客服打开工单详情页时，前端调用 `GET /api/ai/suggest/{ticketId}`

**完整 RAG 流程**：

```
① 客服打开工单
        │
        ▼
② 后端取出 ticket（title + description）
        │
        ▼
③ 调用 Embedding API（DeepSeek / OpenAI）
   将工单文本向量化 → 得到 float[1536]
        │
        ▼
④ PGVector 余弦相似度检索
   SELECT * FROM ticket_embedding
   ORDER BY embedding <=> query_vector
   LIMIT 3（取最相似的3条历史工单）
        │
   相似度 >= 0.75 才使用，否则忽略
        │
        ▼
⑤ 构建增强 Prompt（见下方模板）
        │
        ▼
⑥ 调用 DeepSeek Chat API → 生成回复草稿
        │
        ▼
⑦ 返回给前端展示，客服可编辑后发送
```

**增强 Prompt 模板**：

```
你是一名专业客服代表，请根据用户工单内容，结合历史解决方案，生成一份专业、简洁的回复草稿。

【当前工单】
类型：{category}
优先级：{priority}
标题：{title}
描述：{description}

【历史相似案例】（相似度从高到低）
案例1（相似度{score1}）：
  问题：{similar_description_1}
  解决方案：{solution_text_1}

案例2（相似度{score2}）：
  问题：{similar_description_2}
  解决方案：{solution_text_2}

【要求】
1. 回复语气专业、友好，不超过200字
2. 如果历史案例能直接解决问题，基于历史方案给出具体步骤
3. 如果历史案例无法参考，给出合理的处理思路
4. 末尾附上"如需进一步帮助，请随时联系我们"
```

### 4.3 模型与 API 配置

| 功能 | 使用模型 | API |
|------|---------|-----|
| 工单分类 | deepseek-chat | DeepSeek Chat API |
| 生成回复草稿 | deepseek-chat | DeepSeek Chat API |
| 文本向量化 | text-embedding-3-small | OpenAI Embedding API（或 DeepSeek Embedding） |

> 说明：DeepSeek 的 Embedding API 目前支持 `embedding-2`，维度为 1024；如选用需将 pgvector 建表改为 `vector(1024)`。推荐统一用 OpenAI text-embedding-3-small（1536维），成本低且质量稳定。

**Spring AI 配置示例（application.yml）**：

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/helpdesk
    username: ${DB_USER}
    password: ${DB_PASSWORD}
    driver-class-name: org.postgresql.Driver
  jpa:
    database-platform: org.hibernate.dialect.PostgreSQLDialect
    hibernate:
      ddl-auto: validate   # 用 Flyway 管理表结构，JPA 只做校验
  ai:
    deepseek:
      api-key: ${DEEPSEEK_API_KEY}
      chat:
        options:
          model: deepseek-chat
          temperature: 0.3
    openai:
      api-key: ${OPENAI_API_KEY}
      embedding:
        options:
          model: text-embedding-3-small
```

{ "email": "user@example.com", "password": "123456" }

```json
{ "token": "eyJ...", "role": "USER", "userId": 1, "username": "张三" }
```

### 5.2 工单接口

| 方法 | 路径 | 描述 | 权限 |
|------|------|------|------|
| POST | `/api/tickets` | 提交工单 | USER |
| GET | `/api/tickets` | 查询工单列表（含分页+筛选） | USER/AGENT/ADMIN |
| GET | `/api/tickets/{id}` | 工单详情 | USER(仅自己)/AGENT/ADMIN |
| PATCH | `/api/tickets/{id}/status` | 更新工单状态 | AGENT/ADMIN |
| POST | `/api/tickets/{id}/assign` | 分配工单给客服 | ADMIN |
| POST | `/api/tickets/{id}/rate` | 工单评分 | USER(工单提交者) |

**POST /api/tickets 请求体**：
```json
{
  "title": "无法登录系统",
  "description": "从昨天开始一直提示密码错误，但密码没有改过"
}
```
**响应**（AI分类异步处理，先返回基本信息）：
```json
{
  "id": 101,
  "status": "PENDING",
  "category": null,
  "priority": null,
  "createdAt": "2026-06-29T12:00:00"
}
```

**GET /api/tickets 查询参数**：
```
?status=PENDING&category=TECHNICAL&page=0&size=10&sort=createdAt,desc
```

## 7. 工作分工

> 基于两人均熟悉后端，建议按模块纵向切分，各自全栈负责自己的模块。

### 方案A：模块切分（推荐）

| | 人员 A（基础模块） | 人员 B（AI模块） |
|---|---|---|
| **后端** | 用户认证（注册/登录/JWT）<br>工单CRUD<br>回复模块<br>通知模块（WebSocket）<br>管理员接口（分配+统计） | AI分类异步流程<br>Embedding向量化+存储<br>PGVector相似检索<br>RAG回复建议生成<br>Spring AI配置 |
| **前端** | 登录/注册页<br>用户工单列表+详情<br>管理员页面 | 客服工单列表+处理页<br>AI建议面板组件<br>数据大盘（ECharts） |
| **独立** | MySQL建表+初始化<br>Spring Security配置 | PGVector建表+环境搭建<br>DeepSeek/OpenAI API Key管理 |

**接口对接约定**：
- A负责定义并实现`/api/tickets/{id}`的完整数据结构（包括category/priority字段）
- B在A的工单表上扩展，不修改已有字段
- 双方共用 `UserDetails`，B只读不写用户相关代码

### 方案B：前后端切分

| 人员 A（后端全部） | 人员 B（前端全部） |
|---|---|
| 全部后端接口+AI集成 | 全部Vue页面+组件 |

> 不推荐：后端工作量远大于前端，A会成为瓶颈。

## 8. 开发时间计划（8周）

| 周次 | 人员A | 人员B | 里程碑 |
|------|-------|-------|--------|
| 第1周 | 项目初始化、数据库建表、用户注册登录 | PGVector环境、Spring AI接入DeepSeek、Embedding接口验证 | 环境全部Ready |
| 第2周 | 工单CRUD（提交/列表/详情）、权限控制 | 工单分类异步任务（提交后自动分类）、向量写入流程 | 工单提交→AI分类全链路跑通 |
| 第3周 | 回复模块、工单状态流转、分配接口 | PGVector相似检索、RAG Prompt构建、`/api/ai/suggest`接口 | AI回复建议接口完成 |
| 第4周 | WebSocket通知、管理员统计接口 | 联调AI分类+建议，调优Prompt，补充错误处理 | 后端核心功能完成 |
| 第5周 | 前端：登录/注册、用户工单页面 | 前端：客服处理页面、AI建议面板组件 | 两端前端开发完成 |
| 第6周 | 前端：管理员页面（工单管理+用户管理） | 前端：数据大盘（ECharts）、WebSocket通知前端接入 | 前端全部完成 |
| 第7周 | 前后端联调、Bug修复 | 前后端联调、Bug修复 | 联调完成，系统可用 |
| 第8周 | 测试（功能+边界）、部署文档 | 测试（AI效果评估）、项目报告 | 项目交付 |

### 关键风险与应对

| 风险 | 应对 |
|------|------|
| DeepSeek API限流或不稳定 | 本地缓存AI结果，相同工单不重复调用 |
| PGVector环境配置复杂 | 第1周B优先搞定，其他功能不依赖它 |
| Prompt效果差 | 第4周专门留一周调优，准备5-10条测试用例 |
| 前后端接口对不上 | 第1周末双方对齐接口文档，用Apifox维护 |
