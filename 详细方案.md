## 2. 系统角色与详细用例

### 2.1 角色定义

```
普通用户 (USER)      → 提交工单、查看进度、接收通知
客服人员 (AGENT)     → 处理工单、查看AI建议、回复用户
管理员   (ADMIN)     → 分配工单、查看统计数据、管理用户
```

### 2.2 用例详细说明

#### 普通用户

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 注册/登录 | 邮箱+密码，JWT Token鉴权 | P0 |
| 提交工单 | 填写标题、描述、上传附件（可选）；提交后AI自动分类+优先级 | P0 |
| 查看工单列表 | 我的工单，按状态筛选 | P0 |
| 工单详情 | 查看处理进度、客服回复记录 | P0 |
| 实时通知 | WebSocket推送：工单状态变更、新回复 | P1 |
| 评价工单 | 工单关闭后可打分（1-5星）+ 文字评价 | P2 |

#### 客服人员

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 查看待处理工单 | 只看分配给自己的，按优先级排序 | P0 |
| 查看工单详情+AI建议 | 打开工单时自动触发：①AI生成回复草稿 ②展示相似历史工单 | P0 |
| 回复工单 | 采纳AI草稿或手动编写，发送后通知用户 | P0 |
| 更新工单状态 | PROCESSING → RESOLVED | P0 |
| 查看历史处理记录 | 我处理过的工单统计 | P2 |

#### 管理员

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 分配工单 | 将PENDING工单指派给指定客服 | P0 |
| 数据大盘 | 今日工单量、平均响应时长、各类型占比、AI采纳率 | P1 |
| 用户管理 | 查看用户列表、禁用账号 | P1 |
| 客服管理 | 查看客服工作量、绩效 | P2 |

## 3. 数据库设计

> 所有表统一存放在同一个 PostgreSQL 15 数据库中，结构化数据与向量数据共库，无需维护两个数据源。

### 3.1 初始化

```sql
-- 启用 pgvector 扩展（仅需执行一次）
CREATE EXTENSION IF NOT EXISTS vector;
```

### 3.2 表结构

#### `users` 用户表

```sql
CREATE TABLE users (
    id          BIGSERIAL PRIMARY KEY,
    username    VARCHAR(50)  NOT NULL UNIQUE,
    email       VARCHAR(100) NOT NULL UNIQUE,
    password    VARCHAR(255) NOT NULL,   -- BCrypt加密
    role        VARCHAR(10)  NOT NULL DEFAULT 'USER'
                    CHECK (role IN ('USER','AGENT','ADMIN')),
    status      VARCHAR(10)  NOT NULL DEFAULT 'ACTIVE'
                    CHECK (status IN ('ACTIVE','DISABLED')),
    avatar_url  VARCHAR(255),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `ticket` 工单表

```sql
CREATE TABLE ticket (
    id              BIGSERIAL PRIMARY KEY,
    title           VARCHAR(200) NOT NULL,
    description     TEXT NOT NULL,
    category        VARCHAR(20) CHECK (category IN ('TECHNICAL','BILLING','COMPLAINT','OTHER')),  -- AI填充
    priority        VARCHAR(10) CHECK (priority IN ('LOW','MEDIUM','HIGH','URGENT')),             -- AI填充
    status          VARCHAR(20) NOT NULL DEFAULT 'PENDING'
                        CHECK (status IN ('PENDING','ASSIGNED','PROCESSING','RESOLVED','CLOSED')),
    submitter_id    BIGINT NOT NULL REFERENCES users(id),
    assignee_id     BIGINT REFERENCES users(id),           -- 客服
    ai_classified   BOOLEAN DEFAULT FALSE,                  -- 是否已AI分类
    rating          SMALLINT CHECK (rating BETWEEN 1 AND 5),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    resolved_at     TIMESTAMPTZ
);
```

#### `reply` 回复表

```sql
CREATE TABLE reply (
    id          BIGSERIAL PRIMARY KEY,
    ticket_id   BIGINT NOT NULL REFERENCES ticket(id),
    author_id   BIGINT NOT NULL REFERENCES users(id),
    content     TEXT NOT NULL,
    is_ai_draft BOOLEAN DEFAULT FALSE,   -- 是否来自AI草稿
    ai_adopted  BOOLEAN DEFAULT FALSE,   -- 客服是否直接采纳AI内容
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `notification` 通知表

```sql
CREATE TABLE notification (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT NOT NULL REFERENCES users(id),
    type        VARCHAR(20) NOT NULL
                    CHECK (type IN ('STATUS_CHANGE','NEW_REPLY','ASSIGNED')),
    ticket_id   BIGINT NOT NULL REFERENCES ticket(id),
    message     VARCHAR(500) NOT NULL,
    is_read     BOOLEAN DEFAULT FALSE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `ticket_embedding` 向量表

```sql
CREATE TABLE ticket_embedding (
    id              BIGSERIAL PRIMARY KEY,
    ticket_id       BIGINT NOT NULL UNIQUE REFERENCES ticket(id),
    content_text    TEXT NOT NULL,       -- 原始文本（title + description）
    embedding       vector(1536),        -- text-embedding-3-small 维度
    category        VARCHAR(20),
    solution_text   TEXT,               -- 最终解决方案（resolved后填入）
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- HNSW 向量索引（余弦相似度）
CREATE INDEX ON ticket_embedding USING hnsw (embedding vector_cosine_ops);
```

## 4. AI 功能详细设计

> **设计原则**：所有 AI 调用集中在客服打开工单时的一次同步请求中完成。用户提交工单只写数据库，不触发任何 AI 流程，消除异步耦合。

### 4.1 工单分类（按需触发，同步）

**触发时机**：客服调用 `GET /api/ai/suggest/{ticketId}` 时，若 `ai_classified = false`，在同一请求内先执行分类再生成建议。

**流程**：

```
客服打开工单
    │
    ▼
检查 ticket.ai_classified
    │
    ├── false → 调用 DeepSeek 分类 Prompt（同步等待）
    │              → 更新 ticket.category / priority / ai_classified=true
    │
    └── true  → 跳过，直接进入 RAG 流程
```

**分类 Prompt 模板**：

```
你是一个客服工单分类助手。请根据以下工单内容，返回JSON格式的分类结果。

工单标题：{title}
工单描述：{description}

请严格按照以下JSON格式返回，不要输出任何其他内容：
{
  "category": "TECHNICAL|BILLING|COMPLAINT|OTHER",
  "priority": "LOW|MEDIUM|HIGH|URGENT",
  "reason": "一句话说明分类依据"
}

分类规则：
- TECHNICAL：涉及功能故障、无法登录、系统错误
- BILLING：涉及费用、退款、账单问题
- COMPLAINT：投诉服务态度或处理结果
- OTHER：不属于以上类别
优先级判断：URGENT=服务完全不可用；HIGH=主要功能受影响；MEDIUM=部分功能受影响；LOW=咨询类
```

---

### 4.2 AI 回复建议 + 向量写入（单次同步请求）

**触发时机**：客服打开工单时，前端调用 `GET /api/ai/suggest/{ticketId}`，后端在该请求内按顺序完成所有 AI 操作。

**完整流程（单次同步请求）**：

```
GET /api/ai/suggest/{ticketId}
    │
    ▼
① 从 DB 读取 ticket
    │
    ├── ai_classified=false → 调用 DeepSeek 分类（见 4.1）→ 更新 DB
    └── ai_classified=true  → 跳过
    │
    ▼
② 调用 Embedding API 将工单文本向量化 → float[1536]
    │
    ▼
③ PGVector 余弦相似度检索
   ORDER BY embedding <=> query_vector LIMIT 3
   过滤：相似度 < 0.75 的结果丢弃
    │
    ▼
④ 构建增强 Prompt（工单内容 + 相似历史案例）
    │
    ▼
⑤ 调用 DeepSeek 生成回复草稿
    │
    ▼
⑥ 返回 { draft, similarTickets, category, priority }
```

> **向量写入时机**：向量数据代表"已解决的历史案例"，因此只在工单状态变为 `RESOLVED` 时同步写入 `ticket_embedding`（解决方案文本 + 向量），不在提交时写入。

**增强 Prompt 模板**：

```
你是一名专业客服代表，请根据用户工单内容，结合历史解决方案，生成一份专业、简洁的回复草稿。

【当前工单】
类型：{category}
优先级：{priority}
标题：{title}
描述：{description}

【历史相似案例】（相似度从高到低）
案例1（相似度{score1}）：
  问题：{similar_description_1}
  解决方案：{solution_text_1}

案例2（相似度{score2}）：
  问题：{similar_description_2}
  解决方案：{solution_text_2}

【要求】
1. 回复语气专业、友好，不超过200字
2. 如果历史案例能直接解决问题，基于历史方案给出具体步骤
3. 如果历史案例无法参考，给出合理的处理思路
4. 末尾附上"如需进一步帮助，请随时联系我们"
```

### 4.3 模型与 API 配置

| 功能 | 使用模型 | API |
|------|---------|-----|
| 工单分类 | deepseek-chat | DeepSeek Chat API |
| 生成回复草稿 | deepseek-chat | DeepSeek Chat API |
| 文本向量化 | text-embedding-3-small | OpenAI Embedding API（或 DeepSeek Embedding） |

> 说明：DeepSeek 的 Embedding API 目前支持 `embedding-2`，维度为 1024；如选用需将 pgvector 建表改为 `vector(1024)`。推荐统一用 OpenAI text-embedding-3-small（1536维），成本低且质量稳定。

**Spring AI 配置示例（application.yml）**：

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/helpdesk
    username: ${DB_USER}
    password: ${DB_PASSWORD}
    driver-class-name: org.postgresql.Driver
  jpa:
    database-platform: org.hibernate.dialect.PostgreSQLDialect
    hibernate:
      ddl-auto: validate   # 用 Flyway 管理表结构，JPA 只做校验
  ai:
    deepseek:
      api-key: ${DEEPSEEK_API_KEY}
      chat:
        options:
          model: deepseek-chat
          temperature: 0.3
    openai:
      api-key: ${OPENAI_API_KEY}
      embedding:
        options:
          model: text-embedding-3-small
```

## 7. 工作分工

> **核心原则**：第1周两人一起把数据库表结构和所有接口契约定死，之后各自对着契约独立开发，互相用 Mock 数据填充，最后合并。除第1周外，两人不互相等待。

### 唯一的阻塞点（第1周共同完成）

| 必须一起定义的内容 | 说明 |
|---|---|
| 全部表结构（字段、类型、约束） | 本文档第3节即最终版，双方不得擅自加字段 |
| 全部 API 接口（路径、入参、返参） | 本文档第5节即契约，写入 Apifox 并冻结 |
| DTO 类定义 | 在 `dto/` 包里各自用的 DTO 提前建好，两人都能引用 |

第1周结束后，两人可以完全并行，不再互相阻塞。

---

### 人员 A：用户端 + 管理端（全栈）

**后端**
- 用户注册/登录、JWT 鉴权、Spring Security 配置
- 工单 CRUD（提交、列表、详情、状态流转）
- 回复模块（`POST /api/tickets/{id}/replies`）
- 工单分配接口（`POST /api/tickets/{id}/assign`）
- 管理员统计接口（`GET /api/admin/stats`）
- WebSocket 通知推送

**前端**
- 登录/注册页
- 用户工单列表 + 详情页
- 管理员工单管理 + 用户管理页
- 数据大盘（ECharts）

**依赖 B 的接口**：客服处理页需要 `/api/ai/suggest` → 期间用 Mock 数据代替，B 完成后替换真实调用。

---

### 人员 B：客服端 + AI（全栈）

**后端**
- Spring AI 环境配置（DeepSeek、Embedding API）
- PGVector 环境搭建 + `ticket_embedding` 表
- AI 分类逻辑（`AiClassifier`，被 `AiService` 内部调用）
- Embedding 向量化 + PGVector 相似检索
- RAG Prompt 构建 + 回复草稿生成
- `GET /api/ai/suggest/{ticketId}` 接口（独立 Controller）
- 工单 RESOLVED 时写向量（监听 A 暴露的 `TicketResolvedEvent`，见下方约定）

**前端**
- 客服工单列表页（基于 A 的 `/api/tickets` Mock 数据开发）
- 客服工单处理页 + AI 建议面板组件
- WebSocket 通知前端接入

**依赖 A 的接口**：`/api/tickets` 列表和详情 → 期间用 Mock 数据代替，A 完成后替换真实调用。

---

### 唯一跨人协作点：TicketResolvedEvent

B 需要在工单 RESOLVED 时写向量，但 RESOLVED 的状态变更逻辑在 A 那边。约定：

```java
// A 在 TicketService.updateStatus() 中发布事件（一行代码）
applicationContext.publishEvent(new TicketResolvedEvent(ticket));

// B 在自己的 EmbeddingService 中监听（完全独立，不需要改 A 的代码）
@EventListener
public void onTicketResolved(TicketResolvedEvent event) { ... }
```

这样 A 只加一行，B 完全自主，不需要反复沟通。
