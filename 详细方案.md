## 2. 系统角色与详细用例

### 2.1 角色定义

```
普通用户 (USER)      → 提交工单、查看进度、接收通知
客服人员 (AGENT)     → 处理工单、查看AI建议、回复用户
管理员   (ADMIN)     → 分配工单、查看统计数据、管理用户
```

### 2.2 用例详细说明

#### 普通用户

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 注册/登录 | 邮箱+密码，JWT Token鉴权 | P0 |
| 提交工单 | 填写标题、描述、上传附件（可选）；提交后AI自动分类+优先级 | P0 |
| 查看工单列表 | 我的工单，按状态筛选 | P0 |
| 工单详情 | 查看处理进度、客服回复记录 | P0 |
| 实时通知 | WebSocket推送：工单状态变更、新回复 | P1 |
| 评价工单 | 工单关闭后可打分（1-5星）+ 文字评价 | P2 |

#### 客服人员

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 查看待处理工单 | 只看分配给自己的，按优先级排序 | P0 |
| 查看工单详情+AI建议 | 打开工单时自动触发：①AI生成回复草稿 ②展示相似历史工单 | P0 |
| 回复工单 | 采纳AI草稿或手动编写，发送后通知用户 | P0 |
| 更新工单状态 | PROCESSING → RESOLVED | P0 |
| 查看历史处理记录 | 我处理过的工单统计 | P2 |

#### 管理员

| 用例 | 说明 | 优先级 |
|------|------|--------|
| 分配工单 | 将PENDING工单指派给指定客服 | P0 |
| 数据大盘 | 今日工单量、平均响应时长、各类型占比、AI采纳率 | P1 |
| 用户管理 | 查看用户列表、禁用账号 | P1 |
| 客服管理 | 查看客服工作量、绩效 | P2 |

## 3. 数据库设计

> 所有表统一存放在同一个 PostgreSQL 15 数据库中，结构化数据与向量数据共库，无需维护两个数据源。

### 3.1 初始化

```sql
-- 启用 pgvector 扩展（仅需执行一次）
CREATE EXTENSION IF NOT EXISTS vector;
```

### 3.2 表结构

#### `users` 用户表

```sql
CREATE TABLE users (
    id          BIGSERIAL PRIMARY KEY,
    username    VARCHAR(50)  NOT NULL UNIQUE,
    email       VARCHAR(100) NOT NULL UNIQUE,
    password    VARCHAR(255) NOT NULL,   -- BCrypt加密
    role        VARCHAR(10)  NOT NULL DEFAULT 'USER'
                    CHECK (role IN ('USER','AGENT','ADMIN')),
    status      VARCHAR(10)  NOT NULL DEFAULT 'ACTIVE'
                    CHECK (status IN ('ACTIVE','DISABLED')),
    avatar_url  VARCHAR(255),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `ticket` 工单表

```sql
CREATE TABLE ticket (
    id              BIGSERIAL PRIMARY KEY,
    title           VARCHAR(200) NOT NULL,
    description     TEXT NOT NULL,
    category        VARCHAR(20) CHECK (category IN ('TECHNICAL','BILLING','COMPLAINT','OTHER')),  -- AI填充
    priority        VARCHAR(10) CHECK (priority IN ('LOW','MEDIUM','HIGH','URGENT')),             -- AI填充
    status          VARCHAR(20) NOT NULL DEFAULT 'PENDING'
                        CHECK (status IN ('PENDING','ASSIGNED','PROCESSING','RESOLVED','CLOSED')),
    submitter_id    BIGINT NOT NULL REFERENCES users(id),
    assignee_id     BIGINT REFERENCES users(id),           -- 客服
    ai_classified   BOOLEAN DEFAULT FALSE,                  -- 是否已AI分类
    rating          SMALLINT CHECK (rating BETWEEN 1 AND 5),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    resolved_at     TIMESTAMPTZ
);
```

#### `reply` 回复表

```sql
CREATE TABLE reply (
    id          BIGSERIAL PRIMARY KEY,
    ticket_id   BIGINT NOT NULL REFERENCES ticket(id),
    author_id   BIGINT NOT NULL REFERENCES users(id),
    content     TEXT NOT NULL,
    is_ai_draft BOOLEAN DEFAULT FALSE,   -- 是否来自AI草稿
    ai_adopted  BOOLEAN DEFAULT FALSE,   -- 客服是否直接采纳AI内容
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### `notification` 通知表

```sql
CREATE TABLE notification (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT NOT NULL REFERENCES users(id),
    type        VARCHAR(20) NOT NULL
                    CHECK (type IN ('STATUS_CHANGE','NEW_REPLY','ASSIGNED')),
    ticket_id   BIGINT NOT NULL REFERENCES ticket(id),
    message     VARCHAR(500) NOT NULL,
    is_read     BOOLEAN DEFAULT FALSE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

#### 向量存储

向量存储不再自定义 `ticket_embedding` 业务表，统一使用 Spring AI `PgVectorStore` 标准表 `vector_store`。

当前项目约定：
- 业务 schema 脚本只管理 `users`、`ticket`、`reply`、`notification` 等业务表。
- `vector_store` 表结构暂不由应用启动自动创建，需由数据库初始化脚本、DBA 或后续迁移工具预先准备。
- Spring AI 通过 `VectorStore` API 读写 `vector_store`，业务代码不直接拼写 pgvector SQL。

`vector_store` 逻辑字段：

| 字段 | 说明 |
|------|------|
| `id` | Spring AI 文档主键 |
| `content` | 用于向量化和检索的文本，建议为工单标题 + 描述 + 最终解决方案 |
| `metadata` | JSON 元数据，保存 `ticketId`、`category`、`priority`、`status`、`solution` 等业务字段 |
| `embedding` | 1536 维文本向量，对应 `text-embedding-3-small` |

> 注意：由于当前阶段暂不管理向量表结构，`spring.ai.vectorstore.pgvector.initialize-schema=false`。如果后续启用 Spring AI 自动建表，再改为 `true`。

## 4. AI 功能详细设计

> **设计原则**：所有 AI 调用集中在客服打开工单时的一次同步请求中完成。用户提交工单只写数据库，不触发任何 AI 流程，消除异步耦合。

### 4.1 工单分类（按需触发，同步）

**触发时机**：客服调用 `GET /api/ai/suggest/{ticketId}` 时，若 `ai_classified = false`，在同一请求内先执行分类再生成建议。

> 当前阶段先使用后端本地确定性规则写回 `category`、`priority`、`ai_classified=true`，用于打通接口闭环；后续接入 DeepSeek Chat 后，将该规则替换为分类 Prompt 调用。

**流程**：

```
客服打开工单
    │
    ▼
检查 ticket.ai_classified
    │
    ├── false → 调用 DeepSeek 分类 Prompt（同步等待）
    │              → 更新 ticket.category / priority / ai_classified=true
    │
    └── true  → 跳过，直接进入 RAG 流程
```

**分类 Prompt 模板**：

```
你是一个客服工单分类助手。请根据以下工单内容，返回JSON格式的分类结果。

工单标题：{title}
工单描述：{description}

请严格按照以下JSON格式返回，不要输出任何其他内容：
{
  "category": "TECHNICAL|BILLING|COMPLAINT|OTHER",
  "priority": "LOW|MEDIUM|HIGH|URGENT",
  "reason": "一句话说明分类依据"
}

分类规则：
- TECHNICAL：涉及功能故障、无法登录、系统错误
- BILLING：涉及费用、退款、账单问题
- COMPLAINT：投诉服务态度或处理结果
- OTHER：不属于以上类别
优先级判断：URGENT=服务完全不可用；HIGH=主要功能受影响；MEDIUM=部分功能受影响；LOW=咨询类
```

---

### 4.2 AI 回复建议 + 向量写入（单次同步请求）

**触发时机**：客服打开工单时，前端调用 `GET /api/ai/suggest/{ticketId}`，后端在该请求内按顺序完成所有 AI 操作。

**完整流程（单次同步请求）**：

```
GET /api/ai/suggest/{ticketId}
    │
    ▼
① 从 DB 读取 ticket
    │
    ├── ai_classified=false → 调用 DeepSeek 分类（见 4.1）→ 更新 DB
    └── ai_classified=true  → 跳过
    │
    ▼
② 调用 Spring AI VectorStore 相似检索，内部使用 OpenAI Embedding API 生成查询向量
    │
    ▼
③ PgVectorStore 余弦相似度检索
   topK=3
   过滤：相似度 < 0.75 的结果丢弃
    │
    ▼
④ 构建增强 Prompt（工单内容 + 相似历史案例）
    │
    ▼
⑤ 调用 DeepSeek 生成回复草稿
    │
    ▼
⑥ 返回 { draft, similarTickets, category, priority }
```

> **向量写入时机**：向量数据代表"已解决的历史案例"，因此只在工单状态变为 `RESOLVED` 时通过 Spring AI `VectorStore.add(...)` 写入 `vector_store`（解决方案文本 + 元数据 + 向量），不在提交时写入。

**增强 Prompt 模板**：

```
你是一名专业客服代表，请根据用户工单内容，结合历史解决方案，生成一份专业、简洁的回复草稿。

【当前工单】
类型：{category}
优先级：{priority}
标题：{title}
描述：{description}

【历史相似案例】（相似度从高到低）
案例1（相似度{score1}）：
  问题：{similar_description_1}
  解决方案：{solution_text_1}

案例2（相似度{score2}）：
  问题：{similar_description_2}
  解决方案：{solution_text_2}

【要求】
1. 回复语气专业、友好，不超过200字
2. 如果历史案例能直接解决问题，基于历史方案给出具体步骤
3. 如果历史案例无法参考，给出合理的处理思路
4. 末尾附上"如需进一步帮助，请随时联系我们"
```

### 4.3 模型与 API 配置

| 功能 | 使用模型 | API |
|------|---------|-----|
| 工单分类 | deepseek-v4-flash | DeepSeek OpenAI 兼容 Chat API |
| 生成回复草稿 | deepseek-v4-flash | DeepSeek OpenAI 兼容 Chat API |
| 文本向量化 | text-embedding-3-small | OpenAI Embedding API |

> 说明：项目只引入 Spring AI OpenAI starter。Chat 通过 OpenAI 兼容接口指向 DeepSeek，Embedding 使用 OpenAI `text-embedding-3-small`。这样后端统一使用 Spring AI `ChatModel`、`ChatClient`、`VectorStore` 抽象。

**Spring AI 配置示例（application.yml）**：

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/helpdesk
    username: ${DB_USER}
    password: ${DB_PASSWORD}
    driver-class-name: org.postgresql.Driver
  jpa:
    hibernate:
      ddl-auto: none       # 当前阶段不让应用自动管理表结构
  ai:
    vectorstore:
      pgvector:
        initialize-schema: false
        dimensions: 1536
        distance-type: COSINE_DISTANCE
        index-type: HNSW
    model:
      chat: openai
      embedding: openai
    openai:
      api-key: ${OPENAI_API_KEY}
      chat:
        base-url: https://api.deepseek.com
        api-key: ${DS_API_KEY}
        options:
          model: deepseek-v4-flash
          temperature: 0.3
      embedding:
        options:
          model: text-embedding-3-small
```

### 5.2 工单接口

|------|------|------|------|

```json
{
  "id": 101,
  "status": "PENDING",

  "priority": null,
  "createdAt": "2026-06-29T12:00:00"
}
```

```
?status=PENDING&category=TECHNICAL&page=0&size=10&sort=createdAt,desc
```

```json
{
  "rating": 5,
```

### 5.5 管理员接口

| 方法 | 路径 | 描述 | 权限 |
|------|------|------|------|
| GET | `/api/admin/stats` | 数据大盘统计 | ADMIN |
| GET | `/api/admin/users` | 用户列表 | ADMIN |
| PATCH | `/api/admin/users/{id}/status` | 禁用/启用用户 | ADMIN |

**GET /api/admin/stats 响应**：
```json
{
  "todayTotal": 45,
  "pendingCount": 12,
  "avgResponseMinutes": 18,
  "categoryDistribution": {
    "TECHNICAL": 0.42, "BILLING": 0.28, "COMPLAINT": 0.15, "OTHER": 0.15
  },
  "aiAdoptionRate": 0.67
}
```

指标口径：

| 字段 | 说明 |
|------|------|
| `todayTotal` | 当天 00:00 以后创建的工单数 |
| `pendingCount` | 当前状态为 `PENDING` 的工单数 |
| `avgResponseMinutes` | 工单创建到首次回复的平均分钟数，保留两位小数；无回复时为 0 |
| `categoryDistribution` | 已分类工单按 `category` 分组后的占比，保留两位小数 |
| `aiAdoptionRate` | 回复中 `aiAdopted=true` 的占比，保留两位小数；无回复时为 0 |

**GET /api/admin/users 查询参数**：
```text
?role=USER&status=ACTIVE&page=0&size=10&sort=createdAt,desc
```

参数说明：

| 参数 | 必填 | 说明 |
|------|------|------|
| `role` | 否 | 按角色筛选：`USER` / `AGENT` / `ADMIN` |
| `status` | 否 | 按账号状态筛选：`ACTIVE` / `DISABLED` |
| `page` | 否 | 页码，从 0 开始，默认 0 |
| `size` | 否 | 每页数量，默认 10 |
| `sort` | 否 | 排序字段，默认 `createdAt,desc` |

**GET /api/admin/users 响应**：
```json
{
  "content": [
    {
      "id": 1,
      "username": "zhangsan",
      "email": "zhangsan@example.com",
      "role": "USER",
      "status": "ACTIVE",
      "avatarUrl": null,
      "createdAt": "2026-06-30T10:00:00+08:00",
      "updatedAt": "2026-06-30T10:00:00+08:00"
    }
  ],
  "page": 0,
  "size": 10,
  "totalElements": 1,
  "totalPages": 1,
  "first": true,
  "last": true
}
```

> 用户响应不返回 `password` 字段。`DISABLED` 用户不能继续登录，已有 JWT 也会被鉴权过滤器拒绝。

**PATCH /api/admin/users/{id}/status 请求体**：
```json
{
  "status": "DISABLED"
}
```

**PATCH /api/admin/users/{id}/status 响应**：返回更新后的用户信息，结构同 `UserResponse`。

## 7. 工作分工

> 前后端分离开发，第1周共同定好接口契约后即可独立并行，B 全程对着 Mock 数据开发，第7周统一联调替换。

### 人员 A：后端（全部）

| 模块 | 具体内容 |
|------|---------|
| 基础设施 | PostgreSQL + pgvector 环境、SQL 初始化脚本、Spring Security + JWT |
| 用户模块 | 注册、登录、角色鉴权 |
| 工单模块 | 提交、列表、详情、状态流转、分配 |
| 回复模块 | 发送回复、查询回复列表 |
| AI 模块 | Spring AI 配置、DeepSeek 分类、Embedding + PGVector 检索、RAG 生成建议 |
| 通知模块 | WebSocket 推送、通知已读 |
| 管理员模块 | 数据统计接口、用户列表、账号启用/禁用 |

> 注意：后端工作量大于前端，A 若进度紧张，可把"管理员统计接口"推迟到第6周，不影响 B 的前端进度。

### 人员 B：前端（全部）

| 页面 | 具体内容 |
|------|---------|
| 公共 | 登录/注册、路由守卫、axios 封装、Pinia 状态 |
| 用户端 | 工单列表、提交工单、工单详情 |
| 客服端 | 客服工单列表、工单处理页（含 AI 建议面板、相似工单抽屉） |
| 管理员端 | 工单管理、用户管理、数据大盘（ECharts） |
| 通用 | WebSocket 通知接入、通知 Badge |

B 全程使用 Apifox Mock Server 联调，A 完成对应接口后通知 B 切换真实地址即可。

---

### 第1周必须共同完成的内容（唯一阻塞点）

- 确认数据库表结构（本文档第3节），当前阶段由 SQL 脚本或人工执行准备，应用启动不自动建表
- 将第5节所有接口录入 **Apifox**，开启 Mock，B 即可开始前端开发
- 确定统一的错误响应格式，例如：
  ```json
  { "code": 400, "message": "工单不存在", "data": null }
  ```

## 8. 开发时间计划（8周）

| 周次 | 人员 A（后端） | 人员 B（前端） |
|------|--------------|--------------|
| **第1周** | **共同**：DB 表结构确认与初始化、接口录入 Apifox、项目脚手架 | 同左 |
| 第2周 | 用户认证、工单 CRUD、权限控制 | 登录/注册页、用户工单列表+详情（对接 Mock） |
| 第3周 | 回复模块、工单状态流转、分配接口 | 提交工单页、客服工单列表（对接 Mock） |
| 第4周 | Spring AI 环境、AI 分类、Embedding + PGVector | 客服工单处理页、AI 建议面板组件（对接 Mock） |
| 第5周 | RAG 建议生成、`/api/ai/suggest` 接口、WebSocket | 管理员工单管理、用户管理页（对接 Mock） |
| 第6周 | 管理员统计接口、Prompt 调优、错误处理补全 | 数据大盘（ECharts）、WebSocket 通知接入 |
| **第7周** | **联调**：A 通知 B 逐个接口切换真实地址，修 Bug | 同左 |
| 第8周 | 测试、部署文档 | UI 细节打磨、项目报告 |

**总结**：除第1周（定契约）和第7周（联调）外，两人完全独立，B 不等 A 的任何接口。

### 关键风险与应对

| 风险 | 应对 |
|------|------|
| 接口契约第1周没定清楚 | Apifox 接口一旦录入，改动必须双方确认；B 发现 Mock 数据不够用时及时提 |
| A 后端进度慢，第7周来不及联调 | A 优先保证核心接口（认证 + 工单 CRUD + AI建议），统计接口可推后 |
| B 对接真实接口时字段对不上 | 第1周定好统一 DTO 结构，B 的 TypeScript 类型直接从 Apifox 导出 |
| DeepSeek API 不稳定 | A 写一个 `MockAiService` 实现同一接口，本地开发默认走 Mock |
